## How to Recreate the Project

### Step 1: Scaffold

The project uses the Manus `web-db-user` template (React 19 + Tailwind 4 + Express 4 + tRPC 11 + Drizzle ORM + TiDB). The template provides framework plumbing in `server/_core/`, `client/src/_core/`, and `shared/_core/` — these files are NOT included in this handoff because they are auto-generated by the platform. The template also provides shadcn/ui components in `client/src/components/ui/` — also auto-generated.

**What the template provides (DO NOT recreate):**
- `server/_core/` — Express server, tRPC context, OAuth, JWT, LLM helpers, image generation, voice transcription, notification, SDK
- `client/src/_core/` — useAuth hook
- `shared/_core/` — Error types
- `client/src/components/ui/` — shadcn/ui components (button, card, dialog, dropdown, input, label, select, separator, sheet, skeleton, slider, switch, table, tabs, textarea, toast, tooltip)
- `node_modules/` — installed via `pnpm install`

**What you recreate from these handoff documents:**
- Everything in `server/` (except `server/_core/`)
- Everything in `client/src/` (except `client/src/_core/` and `client/src/components/ui/`)
- Everything in `drizzle/` (schema and relations only — migrations are auto-generated by `pnpm db:push`)
- Everything in `shared/` (except `shared/_core/`)
- All config files in the project root

### Step 2: File Creation Order

Create files in this order to avoid import errors:

### Step 3: Database Migration

After creating `drizzle/schema.ts`, run:
```bash
pnpm db:push
```
This generates migration SQL files and applies them to TiDB.

### Step 4: Install Dependencies

```bash
pnpm install
```

```bash

pnpm test                  # Should show 218 tests passing
pnpm build                 # Should succeed in ~6 seconds
```

## CDN Catalog Data Source

The application's primary data source is a 2.5MB JSON file hosted on CloudFront:

```
https://d2xsxph8kpxj0f.cloudfront.net/310519663317311392/strainscout-md-data/catalog-v8-validated.json
```

This file contains:
- `strains[]` — 2,220 strain objects with name, brand, type, THC/CBD, terpenes, effects, flavors, genetics, prices, dispensary availability
- `dispensaries[]` — 97 dispensary objects with name, city, address, phone, hours, brands, strain count
- `brands[]` — Brand objects with name, strain count
- `metadata` — Last updated timestamp, validation score, version

The catalog is fetched client-side by the `useCatalog()` hook and cached in a module-level singleton. It is NOT stored in the database — the database only stores user-generated data (votes, comments, alerts, partner claims).

## Known Issues (23 findings)

See `strainscout_fix_strategy.md` for the complete fix strategy with code snippets and test plans. The top 5 most impactful:

1. **PriceUpdateForm cache not invalidated** — Partner submits a price, "Recent Submissions" list doesn't update until page refresh
2. **Slugify function duplicated 6 times** — Same logic in 4 files + 2 inline usages
3. **4 analytics events defined but never called** — trackPriceCompared, trackDispensaryClicked, trackOutboundLinkClicked, trackMapInteracted
4. **Race condition in claimDispensary()** — SELECT-then-INSERT; UNIQUE constraint prevents corruption but error leaks raw MySQL
5. **919KB main bundle chunk** — No manual chunk splitting configured
